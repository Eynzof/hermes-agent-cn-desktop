## 1. TL;DR

Hermes Desktop v2 是 Hermes Agent CN 的 **Tauri 重写版桌面客户端**，替换原 Electron 实现（v1）。它对接 Hermes Dashboard 后端，给中文社区用户提供一个比 Web 更原生、比 Electron 更轻的桌面 Agent 操作台。本阶段不追多用户、不追上线时间点，**以"本人日常重度使用且挑不出毛病"为验收标准**。

---

## 2. 背景

### 2.1 仓库现状

- 当前仓库 `hermes-agent-cn-desktop`：Tauri v2 + React + pnpm workspace，已完成基础架构搭建（见 [README](../../README.md) 和 [CLAUDE.md](../../CLAUDE.md)）
- 老仓库 `hermes-desktop`（fathah 仓 fork）：Electron 实现，停止维护，作为参考
- 后端：`hermes-agent-cn` 内置 Dashboard（桌面端 managed runtime 默认端口 9120），桌面端**只做外接层**，尽量不改 Dashboard 代码
- 原型：`hermes-cn-ui-prototypes` Astro 项目，36 个高保真页（深色 + 中英双语），已经画完所有主要场景

### 2.2 历史 vs 当前策略

| 阶段 | 决策 | 结果 |
|------|------|------|
| 早期 | 「先 Web，再桌面」（hermes/docs/PRD.md 主文档原话） | 形成了 Web Workbench PRD 包 |
| 中期 | 「桌面分发成本高，**重心转 Web**」（v2/docs/web-workbench-prd/README） | Web 端推进了一段 |
| 当前 | **回到「桌面优先」**：Tauri v2 重写已落地，做桌面端 PRD 来对齐 | 本文档 |

> 此前没有专门写过桌面端 PRD，本文档是第一份。

### 2.3 用户基础

- 本人 + 身边开发者朋友（核心使用群）
- 中文社区潜在用户：**已有 400 人投票表达使用意向**，按 Hermes Agent 中文社区当前关注度估算潜在用户**万级**
- 但本阶段 PRD **不为外部用户做承诺**，验收只看本人

## 3. 目标与非目标

### 3.1 v2 目标

1. **跑得起来**：本人在 macOS / Windows 上每日使用，端到端无阻塞
2. **替代 Electron 老桌面**：原 Electron 版能做的，新桌面端都能做（功能不退步）
3. **比 Web 更顺**：发挥桌面端原生优势——拖拽、通知、菜单栏、托盘、自动更新
4. **挑不出毛病**：本人对每个核心路径都挑不出体验缺陷，整体感觉成熟

### 3.2 非目标（v2 不做）

| 推后项 | 理由 |
|--------|------|
| 日报功能（原型 21 / 21a） | 需要后端新增聚合任务 + 数据积累期；v2 优先核心闭环 |
| Webhooks / 外部触发器（原型 11 子页暗示） | 后端要起 webhook 端点，v2 先做 Cron 例程一种 |
| 深链接 `hermes://` URL scheme | 等 Web 端公网部署成熟后再做 |
| Skills 画廊视图（09a） | 09 表格视图已够用，砍掉减少 code path |
| Profiles 多视图（20 详情页 / 20a 卡片） | 只保留 20b 列表+详情侧栏 |
| 任务详情折叠态独立路由（03b–06b） | 改为同一路由内右侧面板可隐藏，不开新 URL |
| 多用户 / 多机部署 / 企业管控 | 本阶段桌面是单人本地工具 |
| 完整国际化 | 中文优先；英文跟随原型已有的 lang 切换即可，但**不为非中文用户做适配** |

## 4. 关键决策（与依据）

> 这 9 条决策对应 § 5 范围、§ 7 IA、§ 8 里程碑的取舍。

| # | 决策 | 依据 |
|---|------|------|
| D1 | **「项目」就是 workspace 的 UI 化**，不增加后端抽象 | 后端只有 workspace 概念，加一层抽象会跟 `hermes profile` 命名冲突，纯 UI 包装最快 |
| D2 | **「工作室 / 档案」就是 Hermes profile 的 UI 包装**，不增加后端 | 同上，对齐 `hermes profile switch` 即可 |
| D3 | **顶导留独立入口，侧栏冗余砍掉**：Cron 只在「自动化」、Models 只在顶导第 6 项 | 双入口浪费 code path，单入口最清晰 |
| D4 | **v2 只做 Cron 例程**，不做触发器 / Webhooks | 后端没起 webhook，先把例程做扎实 |
| D5 | **任务详情折叠态用「右侧面板可隐藏」实现**，不开独立路由 | 4 个独立路由意味着 4 套 code path，UI 状态切换足够 |
| D6 | **Skills 留 09 表格、Profiles 留 20b 列表+侧栏**，其它变体砍 | 单一布局更易维护；切视图属偏好功能，v2 不做 |
| D7 | **日报 v2 不做** | 见 § 3.2 |
| D8 | **原生能力 v2 包含**：拖拽附件、本地通知、原生菜单栏；**深链接推后** | 前三个 Tauri 原生支持成本低、体验提升明显；深链接需要协调 Web 端 |
| D9 | **首次启动用轻引导**：登录页 + 工作台 Preflight 清单，不强制 wizard | 本人 + 朋友是熟手用户，强 wizard 反而烦 |

## 7. 信息架构（简版，详细到 02）

```
顶导（6 项，左到右）：
  工作台 | 项目 | 能力 | 自动化 | 可观测 | 模型

每个顶导项的侧栏 / 子页：
  工作台：侧栏（项目快捷 + 进行中会话） + Composer 主区
  项目：侧栏（项目快捷 + 视图切换） + 列表/详情
  能力：侧栏（Skills / MCP，Cron 已移除） + 主区
  自动化：侧栏（例程；触发器/Webhooks disabled） + 主区
  可观测：侧栏（日志 / 健康；Models 已移除） + 主区
  模型：直接进供应商列表（无子页）

辅助 / 全局：
  ⌘K       → 命令面板（modal）
  ⌘?       → 设置中心
  右上角   → 当前 profile / 用户菜单
  托盘     → 任务进度 + 通知
  菜单栏   → File / Edit / View / Window / Help
```

完整路由、深链接、键位绑定到 `02-information-architecture.md`。

## 8. 核心用户场景

> 不写「用户故事 As-a-When-I-Want-So-That」八股，直接走真实路径。

### 场景 A · 起一个开发任务（最高频）

1. 打开桌面 app（系统托盘 / 程序坞 / 上次开过的窗口恢复）
2. 工作台首页：写提示词 → `@file` 拖个文件进 Composer → 选工作区 / 模型 / 审批策略
3. 右侧 Preflight 8 项全绿 → `⌘↵` 发送
4. 跳到任务详情三栏：左侧栏看会话列表、中间看 Chat Timeline、右侧默认显示文件 Diff
5. 切右侧 Tab：终端看 shell 输出 / 产物看生成的文件 / 日志看出错堆栈
6. 高风险操作弹审批卡 → 批准 → 继续
7. 任务跑完后台时 → 本地通知 → 点回来看产物 → 下载 / 复制

### 场景 B · 跨会话回看 + 命令面板（次高频）

1. `⌘K` 唤起命令面板
2. 输入关键字 → 切到「会话」过滤
3. 跳到目标会话 → 看 Chat + 产物

### 场景 C · 管理 Skills（中频）

1. 顶导 → 能力 → Skills
2. 表格里启用/禁用某个 Skill
3. 切「我的」Tab → 进入 09b 编辑器 → 改 YAML / JSON → 测试运行 → 保存

### 场景 D · 排查健康问题（中频）

1. 工作台顶部健康网格亮红 → 点击 → 跳到可观测 / 健康页
2. 看是哪个探针挂了，按 smart hint 跳到设置 / 凭证 / Models 修复
3. 修完回健康页手动刷新

### 场景 E · 切 profile / workspace（低频但关键）

1. 右上角用户菜单 → 选档案
2. 跳到设置 → 档案（20b）→ 列表选另一个 profile → 详情面板「切换」按钮
3. 整个 app 切换上下文（profile 切了之后所有 API 走新 profile 的 token）
